# PolicyLens

PolicyLens is a multi-tenant AI-powered expense compliance platform built with Next.js and Supabase.

It allows organizations to:

- Upload and activate policy PDFs.
- Parse employee receipts (image or PDF).
- Compare claims against active policy clauses using vector search.
- Auto-approve, flag, or reject claims with explainable reasoning.
- Let admins review/override decisions and export approved claims to finance systems.
- Track policy gaps from assistant queries and generate draft clause text.

## Project Goals

PolicyLens is designed to solve 4 enterprise expense problems:

- Reduce policy leakage from manual, inconsistent approvals.
- Explain decisions with policy references and confidence.
- Keep strict tenant isolation across organizations.
- Preserve reliability under AI quota/rate-limit outages using deterministic fallbacks.

## Core Features

## End-To-End Business Flows

### 1. Registration And Onboarding

1. User registers via API.
2. DB trigger creates profile row.
3. User creates org (becomes admin) or joins via invite code.
4. Middleware enforces onboarding completion before admin can use admin screens.

### 2. Policy Ingestion

1. Admin uploads policy PDF.
2. File is validated by magic bytes and stored in Supabase Storage (policy-docs bucket).
3. PDF text is chunked.
4. Chunks are embedded to 768-d vectors.
5. policy_documents and policy_chunks rows are inserted under organization scope.

### 3. Claim Submission And Decisioning

1. Employee uploads receipt and purpose.
2. Receipt is stored in receipts bucket.
3. Extraction pipeline runs with fallbacks:
	 - local extraction
	 - PDF text extraction
	 - Gemini OCR
	 - best-effort Gemini OCR
	 - final local merge fallback
4. Active policy chunks are fetched with vector search.
5. Limits/statistics/duplicate checks are computed.
6. Verdict generated by AI (or deterministic fallback on timeout/quota).
7. Claim is inserted with confidence, requires_review, and status.
8. Notifications are sent to employee and admins depending on verdict.

### 4. Admin Review

1. Admin opens flagged claims.
2. Admin approves/rejects claim.
3. If decision differs from AI verdict, verdict_feedback is logged for future calibration.
4. Employee receives manual review notification email.

### 5. Export

1. Admin maps categories to GL codes.
2. Admin exports approved claims in target format.
3. Export output can be consumed by finance process or imported into ERP.

## Security And Multi-Tenancy Model

### Access Control

- Middleware checks auth, organization membership, and role.
- /admin routes are admin-only.
- /employee routes redirect admins away.
- Public access only to login/register/request-access pages.

### RLS Isolation

Supabase RLS is enabled on core tables. Policies ensure:

- Employees can only view/insert own claims.
- Admins can manage claims/policies/chunks/limits/mappings only inside their organization.
- Org data cannot cross tenant boundaries.

### Defensive Server-Side Checks

- Server routes repeat role/org checks even with middleware.
- API filters by organisation_id on reads/writes.
- File uploads validated using magic bytes.
- Signed URLs used for private asset access.

## Environment Variables

Create .env.local in project root.

Required variables:

- NEXT_PUBLIC_SUPABASE_URL
- NEXT_PUBLIC_SUPABASE_ANON_KEY
- SUPABASE_SERVICE_ROLE_KEY
- GEMINI_API_KEY

Optional variables:

- RESEND_API_KEY
- SENDER_EMAIL
- CRON_SECRET
- ENABLE_TESSERACT_OCR

Detailed behavior:

- If RESEND_API_KEY is missing, email sends are mocked to console and app still runs.
- If CRON_SECRET is set, /api/cron/digest requires Authorization: Bearer <CRON_SECRET>.
- ENABLE_TESSERACT_OCR defaults to enabled unless explicitly set to false.

Example .env.local:

```env
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
GEMINI_API_KEY=your-gemini-key

RESEND_API_KEY=your-resend-key
SENDER_EMAIL=alerts@yourdomain.com
CRON_SECRET=replace-with-random-secret
ENABLE_TESSERACT_OCR=true
```

## AI And Extraction Architecture

### Receipt Extraction Pipeline

The backend uses a resilient sequence to avoid hard failures:

1. Local extraction with timeout.
2. PDF text extraction when file is PDF.
3. Gemini OCR extraction.
4. Gemini best-effort OCR fallback.
5. Local extraction merge fallback.
6. Manual override fields always take precedence if supplied.

### Policy Retrieval Pipeline

1. Build semantic query from category + business purpose.
2. Embed query to 768 dimensions.
3. Call match_policy_chunks RPC scoped by organisation.
4. If vector result is empty, fallback query retrieves chunks from active policy documents directly.

### Verdict Pipeline

Inputs considered:

- receipt fields
- policy chunk context
- spend limits
- baseline z-score context
- duplicate detection signal
- previous rejection context
- admin override history

Output:

- verdict: approved | flagged | rejected
- reason
- policy_reference
- confidence

Fallback behavior:

- Gemini timeout or provider failure falls back to deterministic fast verdict.

### Embeddings

- Target vector size: 768.
- Provider dimensions are normalized to 768 when needed.
- Local deterministic embedding fallback used when API unavailable.

## Troubleshooting

### Symptom: Local dev route crashes with missing .next/dev manifests or vendor chunks

Common errors:

- Cannot find module ./vendor-chunks/@supabase.js
- ENOENT .next/dev/routes-manifest.json
- Cache rename ENOENT under .next/dev/cache/webpack

Cause:

- Corrupted or partially invalidated .next/dev cache, often after runtime/config churn.

Fix:

```powershell
cd "d:\antigravity\expense auditor\policylens"
if (Test-Path ".next\dev") { Remove-Item -Recurse -Force ".next\dev" }
```

Then restart dev server.

If issue persists:

1. Stop all running next dev processes.
2. Delete full .next folder.
3. Run npm install.
4. Start dev again.

### Symptom: OCR extraction weak or slow

Checks:

- Ensure ENABLE_TESSERACT_OCR is not false.
- Confirm uploaded image quality and orientation.
- Use manual merchant/amount/category fields for difficult receipts.

### Symptom: Gemini quota errors (429)

Behavior:

- System applies cooldown and fallback paths.
- Claims should still complete with deterministic verdicting.

Action:

- Provision quota/billing for Gemini in production.
- Keep fallback behavior enabled for resilience.

### Symptom: No policy comparison used during analysis

Checks:

1. Verify at least one policy is active.
2. Confirm policy_chunks were inserted for active documents.
3. Confirm organization_id is correctly set on policy docs/chunks and profile.
